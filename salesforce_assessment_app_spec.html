Application Specification

Salesforce
Assessment Builder

A four-module pipeline that connects to a Salesforce org, extracts metadata across all API surfaces, generates structured assessment content via LLM prompts, and formats the results into 18 shareable deliverables.

Version1.0 — Draft
Deliverables18 report types
API SurfacesMetadata, Tooling, SOQL, REST
LLMAnthropic Claude (claude-sonnet-4-5)

§1 System Architecture

The application is structured as a four-stage sequential pipeline. Each stage is an independent module with a well-defined input contract, output schema, and failure boundary. Stages persist their outputs to a local job store so any stage can be re-run in isolation without re-extracting from Salesforce.

1
Org Connection & Authentication

OAuth 2.0 connected app flow or username/password + security token. Returns a session token, instance URL, and org metadata (edition, API version, org ID). Writes session.json to the job store.

2
Metadata Extraction

Parallel extraction across Metadata API, Tooling API, and SOQL/REST. Each extractor produces a typed JSON payload. Outputs are merged into a single org_snapshot.json file keyed by metadata category.

3
Prompt Engine

A registry of 18 prompt templates — one per deliverable. Each template receives a subset of the org snapshot as context. Claude processes templates sequentially or in parallel (configurable). Outputs are raw LLM responses stored as responses/{deliverable_id}.json.

4
Output Formatting

Formatters transform raw LLM responses into final deliverable formats: HTML report, PDF, DOCX, or JSON. A template renderer applies branding, table of contents, and section structure. Final files are written to outputs/{deliverable_id}/.

Job Store Structure

All intermediate and final artefacts are persisted under a single job directory, identified by a UUID generated at connection time.

jobs/
  {job_id}/
    session.json          ← M1 output
    org_snapshot.json     ← M2 output (full merged payload)
    raw/
      metadata_api/       ← per-type XML/JSON from Metadata API
      tooling_api/        ← per-object JSON from Tooling API
      soql/               ← per-query JSON from SOQL/REST
    responses/
      {deliverable_id}.json  ← M3 raw LLM output per deliverable
    outputs/
      {deliverable_id}/      ← M4 formatted files
        report.html
        report.pdf
        report.docx

M2 Metadata Extraction

Module 2 runs parallel extraction across three API surfaces. Each extractor is isolated — a failure in Tooling API extraction does not block Metadata API extraction. All raw responses are persisted before any processing occurs.

Extractor A — Metadata API

Uses listMetadata() to enumerate components, then retrieve() to fetch full definitions. The retrieve call returns a ZIP which is unpacked into raw/metadata_api/.

Extraction Configuration

Rate limiting & chunking

Metadata API retrieve calls are limited to 10,000 components per request. Chunk large orgs automatically. Tooling API: max 2,000 rows per SOQL call — paginate using nextRecordsUrl. SOQL: use Bulk API 2.0 for queries expected to return >10,000 rows (EventLogFile, large custom objects).

Apex class body retrieval

Apex class source is NOT included in standard Metadata API retrieve for code quality analysis. Use Tooling API: SELECT Body FROM ApexClass WHERE Id = '{id}' per class. Cache locally — do not re-fetch if already in job store.

M3 Prompt Engine

Module 3 maintains a registry of 18 prompt templates. Each template specifies which subset of the org snapshot it consumes, the system prompt context, the expected output schema, and the Claude model parameters.

Prompt Template Interface

interface PromptTemplate {
  id: string;                    // e.g. "data_model_analysis"
  deliverable: string;           // Human label
  category: string;              // Foundation | Logic | etc.
  contextSlices: string[];       // Keys from org_snapshot.json
  maxContextTokens: number;      // Truncation limit for context
  systemPrompt: string;          // Role + output format instructions
  userPromptTemplate: string;    // Handlebars template with {{variables}}
  outputSchema: JSONSchema;      // Expected structure of LLM response
  model: string;                 // "claude-sonnet-4-5"
  maxTokens: number;             // Response limit
  temperature: number;           // 0.2–0.4 for analytical, 0.6 for exec
  streaming: boolean;            // Stream to UI in real-time
}

Context Injection Strategy

The org snapshot can be large. Each prompt template declares which context slices it needs — only those slices are serialised and injected. Slices are further summarised if they exceed maxContextTokens.

Context summarisation rules

Apex class bodies: truncate to first 200 lines + last 50 lines per class. SOQL results: include first 500 rows, append count of total rows. Dependency graph: include only components with >3 dependents. Profile/PermSet: include only non-default permissions.

Execution Order & Dependency DAG

Prompts with no dependencies run in parallel (Wave 1). Synthesis prompts that depend on previous outputs run sequentially (Wave 2 and 3).

Wave 1 (parallel):
  metadata_inventory, data_model_analysis, data_quality_report,
  security_governance, automation_review, reports_dashboards,
  integration_analysis, code_quality, test_coverage,
  change_history, team_evaluation, tech_impact, architectural,
  licensing_analysis

Wave 2 (after Wave 1):
  tech_briefing (needs: code_quality, test_coverage, automation_review)

Wave 3 (after Wave 2):
  executive (needs: all Wave 1 + Wave 2)
  customer_report (needs: executive + 6 core outputs)
  engagement_plan (needs: executive)

Error Handling & Retry

LLM failure strategy

On API error or invalid JSON response: retry up to 3 times with exponential backoff (2s, 8s, 32s). On third failure: store error state in responses/{id}.json with "status": "failed". Downstream dependent prompts skip gracefully. Partial reports are generated with a failure notice in the relevant section.

Formatter Interface

interface Formatter {
  deliverableId: string;
  format: 'html' | 'pdf' | 'docx' | 'json' | 'csv';

  // Validate LLM output matches expected schema
  validate(rawResponse: object): ValidationResult;

  // Transform LLM JSON → template-ready data
  transform(rawResponse: object, session: Session): TemplateData;

  // Render to final file
  render(templateData: TemplateData, outputPath: string): Promise<void>;
}

PDF Generation Pipeline

1
Render HTML in memory

Apply Handlebars template with LLM data. Embed all CSS inline. Base64-encode any images (org logo, charts).

2
Launch Puppeteer

puppeteer.launch({ headless: 'new' }). Navigate to data:text/html URI or temp file. Wait for fonts to load.

3
Generate PDF

page.pdf({ format: 'A4', margin: { top: '20mm', bottom: '20mm', left: '20mm', right: '20mm' }, printBackground: true })

4
Write to job store

Save to outputs/{deliverable_id}/report.pdf. Emit completion event to job queue. Update job status in SQLite.

§6 Security & Credential Handling

Access tokens

Access tokens are stored in memory only during the active session. They are never written to disk in plaintext. The job store records only the org ID and instance URL — not the token. Refresh tokens (if used) are encrypted at rest using AES-256 with a key derived from the operator's secret.

Org data handling

The org snapshot contains configuration metadata and SOQL query results. It may include field names and record counts but should never include actual record data (PII, business data). Data Quality extraction queries null rates only — not record content. The job store should be treated as sensitive and access-controlled accordingly.

LLM data transmission

Org metadata is sent to the Anthropic API as prompt context. Before deployment, obtain explicit customer consent for this data flow. Consider a data processing addendum. For highly sensitive orgs, a self-hosted LLM alternative (e.g. Claude via AWS Bedrock in a VPC) should be offered.

Minimum required permissions

The assessment should connect as a dedicated integration user (not a System Administrator) with only the permissions required. A Permission Set defining the minimum access is provided as a separate artefact (AssessmentRunner.permissionset-meta.xml).
