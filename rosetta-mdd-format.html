The MDD Template
Four components. One consistent structure.

Every description should answer four questions in sequence. Keep the total under 500 characters — brevity forces clarity and reduces AI token usage. Not every component applies to every item; include what is relevant and omit what is not.

Business reason / capability this supports. How it is populated or triggered. Where it is consumed — report, flow, integration, UI. What NOT to use it for, or known limitations (if applicable).
Business reason
Which capability, process, or regulatory requirement drives this item's existence. This is the most important component — start here.
Population / trigger
Is it manually entered by a user, set by a flow, calculated by Apex, or imported via an integration? Name the specific automation if applicable.
Named consumers
Name the specific report, dashboard, flow, Apex class, or external system that reads this. "Used in reporting" is not acceptable — name it.
Exclusions / limits
If a near-identical standard field exists but is not used, say so and explain why. Note deprecated uses, edge cases, or fields that appear similar but serve different purposes.
The Core Rule
Document the Why — not the What.
✕ What (avoid)

This field stores the annual revenue of the account. It is a currency field on the Account object.

✓ Why (target)

Captures total contracted ARR at point of close. Populated automatically by the ARR_Rollup flow when an opportunity stage changes to Closed Won. Consumed by the MonthlyExecDashboard. Not used for billing — see Custom_Billing_Amount__c for invoiced figures.

Writing Principles
Six rules for descriptions that work for humans and AI alike.
P — 01
Plain language
Active voice. Short sentences. Write for someone onboarding six months from now who has no context. Aim for a reading age of 12. If you can cut a word, cut it.
P — 02
No assumed knowledge
If a standard object is being reused in an unusual way, or if your org has a bespoke process, state it explicitly in every affected description. Never assume the reader knows.
P — 03
Approved acronyms only
Use acronyms standard in Salesforce and your industry — ARR, CPQ, MRR, SLA. Spell out internal shorthands the first time. If in doubt, ask ChatGPT whether the acronym is widely understood.
P — 04
Under 500 characters
Brevity forces clarity and reduces AI prompt token usage. If you need more detail, the description belongs in a linked process diagram, not in the field itself.
P — 05
Name your consumers
Unnamed consumers add no impact-analysis value. Name the specific dashboard, flow, integration, or Apex class. If it feeds three things, name all three.
P — 06
Document the negatives
If a standard field is deliberately unused, say so in its description. Silence is ambiguous; ambiguity causes rollbacks. Note deprecated uses, replaced processes, and anything that looks functional but is not.
Templates by Metadata Type
Copy, adapt, and fill in the blanks.

These templates cover the most common metadata types. The slots in amber italics are placeholders — replace them with the specific information for each item.

Object
Represents business entity in the capability / module process. Created when trigger or user action. Owned by team. The standard Object is used / not used because reason.
Custom field
Stores data for business reason. Set by user / flow / integration when condition. Consumed by named report / flow / system. Standard field not used / known exclusion if applicable.
Flow / automation
Triggered when event / condition. Purpose: business outcome. Updates or creates named records / fields. Replaces previous process or manual step. Owner: team.
Validation rule
Prevents bad data state on Object. Required because business or regulatory reason. Introduced in project / sprint. Exceptions handled by bypass mechanism or permission set.
Apex class / trigger
Handles operation that cannot be done declaratively because reason. Called by trigger / batch / API. Integrates with external system if applicable. Test class: ClassName_Test.
Permission set
Grants team / persona access to feature / object / field set. Assigned when role or process condition. Do not combine with conflicting permission set because reason.
Where to Start
HOW → WHAT → WHY — the Rosetta documentation sequence.

Don't try to document the entire org at once. Pick one capability area, map the process, identify the metadata it touches, then write the WHY descriptions for those items. Work in bounded sprints and build coverage iteratively.

HOW
Map the process
Build a simple process diagram showing how users and systems interact to get the job done. This is the context everything else hangs from. Use UPN notation — simple activity boxes, no complex modelling language.
WHAT
Identify the metadata
Walk each process step and ask: "What Salesforce metadata delivers this step?" Objects, fields, flows, validation rules, permission sets — link them to the process steps they support. The same item often appears in multiple steps.
WHY
Write the description
For every metadata item you identified, fill the description field using the MDD template. This is where institutional knowledge is preserved — why the decision was made, what it replaced, and what it must not be confused with.
AI Readiness
Writing for the machine
Four rules that make your descriptions AI-safe.

Agentforce, ElementsGPT, and future AI agents will read your descriptions to decide what to change, reuse, or create. These rules ensure the signal is clean.

No nuance or euphemism
Write what you mean directly. "This field is no longer activated" should be "We do not use this field. We created Custom_Status__c to support the automation logic in Flow_OrderProcess."
Spell out your context
AI does not know your org's history, your architectural decisions, or why you reused a standard object. State it explicitly. Every time. In every affected description.
No internal-only acronyms
AI has been trained on public content — not your internal glossary. Use acronyms that are standard in Salesforce and your industry. If you invented it, spell it out.
Every word is literal
AI reads descriptions as facts. If a field description says it is used for billing, AI will treat it as available for billing workflows — even if that use was deprecated two years ago.
Quality Checklist
Seven checks before a description is complete.

Run every description through this checklist before publishing. If any item fails, revise before moving on.

States the business reason — not just the data type
The description explains why this metadata exists, not what kind of field it is.
Names specific consumers — not "used in reports"
At least one named dashboard, flow, integration, or process is referenced.
Active voice and sentences under approximately 20 words each
Passive constructions obscure meaning. Short sentences are processed more reliably by AI.
No internal acronyms without definition
Any shorthand not standard in Salesforce or the wider industry is spelled out in full.
Unused standard fields or deprecated uses are noted
If the standard field is not used, the description says so and explains why a custom alternative exists.
Total length under 500 characters
If more detail is needed, it belongs in a linked process diagram — not in the description field.
No assumed knowledge — readable by someone new to this org
A consultant seeing this metadata for the first time would understand its purpose, context, and constraints without asking anyone.
